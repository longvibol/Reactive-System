Noted: 


Changing the order of those @ExceptionHandler methods is usually not the issue.


In Spring, handler selection is based primarily on the exception type, not the method order inside the same @RestControllerAdvice.

So in your class:

handleBadRequest(BadRequestException.class) handles BadRequestException

handleConflict(DataIntegrityViolationException.class) handles DataIntegrityViolationException

handleValidation(WebExchangeBindException.class) handles WebExchangeBindException

handleGeneric(Exception.class) is the fallback for everything else

That means Exception.class will not steal the others just because it appears earlier or later. Spring will choose the more specific match.

Recommended arrangement

Even though order usually does not control matching, the clean arrangement is:

1- most specific exceptions

2- framework/business exceptions

3-generic fallback last

@ExceptionHandler(WebExchangeBindException.class)
    public Mono<ProblemDetail> handleValidation(WebExchangeBindException ex, ServerWebExchange exchange) {
        log.warn("Validation error. path={}", exchange.getRequest().getPath(), ex);

        String message = "Validation failed.";
        if (!ex.getAllErrors().isEmpty() && ex.getAllErrors().get(0).getDefaultMessage() != null) {
            message = ex.getAllErrors().get(0).getDefaultMessage();
        }

        return Mono.just(problem(exchange, HttpStatus.BAD_REQUEST, "Validation error", message, "/errors/validation-error"));
    }
